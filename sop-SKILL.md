# 产品回撤异常监测技能 (V1.2)

本技能实现对持仓产品的回撤数据自动采集、风险事件触发判断、投研深度分析、客户沟通话术输出、自动化飞书推送以及事件工作流状态更新的全链路自动化。

---

## 🧭 使用场景与限制 (Usage Scenarios)

✅ **USE when:**
- 需要自动扫描和监测持仓产品的回撤异常。
- 需要针对产生回撤异常的产品自动撰写深度投研归因分析报告。
- 需要生成对客陪伴、情绪安抚的沟通参考话术,并分发推送到飞书群。
- 需要对预警系统的事件工作流执行状态回传推进。

❌ **DON'T use when:**
- 用户仅想查询某只基金的实时净值或基本资料(应使用其他产品查询技能)。
- 需要修改预警规则本身的触发阈值。
- 事件大类不属于"售后"或监测类型不属于"产品回撤"(如产品止盈、合规事件等)。

---

## 🧭 核心业务规约 (Business Policy)

- **数据源一致性**:读取外部评价报告时,获取的资料必须要与触发事件实际发生的时间在时间戳上保持一致,严禁跨日错配。
- **工作流闭环**:若通过 API 模式运行,分析话术生成并推送成功后,必须自动向系统回传工作流状态,完成流程闭环。
- **自动化流控托管**:执行过程中的多级异常降级重试、结构化执行日志输出,已完全由主控脚本底层自动接管并规范输出,无需 Agent 顶层手动格式化记录。

## 📋 自动化工作流执行步骤 (Workflow Execution Steps)

### Step 1: 获取触发事件及相关产品
本步骤支持双数据源模式，**默认使用本地文件模式**。当用户明确发出"使用API"、"远程模式"或类似指令时，系统才切换为 API 远程查询模式。

#### 选项 A：本地文件模式 (默认)
读取本技能目录下的 `data/event_record.xlsx`。检索监测事件为 **"产品回撤事件"** 且触发状态为 **"已触发"** 的最新产品。
* **数据来源**：`data/event_record.xlsx`
* **业务逻辑属性要求**：监测事件名称、监测事件描述、项目名称、产品代码、触发状态。
* **触发条件**：仅当存在已触发事件时才继续执行后续步骤；无触发事件时输出 `"今日无已触发预警事件"` 并终止工作流。

#### 选项 B：API 查询与远程模式 (用户明确指令触发)
当用户在运行指令中明确声明"使用API"或"远程模式"时，系统调用 `watchlist_query.py` 获取未处理的高风险预警事件列表。
* **运行指令**：
  ```bash
  python3 scripts/watchlist_query.py --level HIGH --big-class 售后
  ```
* **输出内容**：返回未处理预警事件列表 of JSON 数组（每项包含事件 ID `id`、产品代码 `productId`、事件名称 `tag`、事件描述 `reason`）。

* **⚠️ 降级策略**：API 脚本执行失败时自动重试 3 次（间隔 5 秒），仍失败则自动降级至本地 Excel 模式（选项 A），并在日报中标注数据源为"降级-本地模式"。

* **逻辑跳转**：
  - **IF 存在已触发/未处理事件**：记录事件信息与事件 ID，继续 Step 2。
  - **IF 无触发事件**：跳过工作流，输出 `"今日无已触发预警事件"`。

---

### Step 2: 采集产品数据
> 💡 **数据源说明**:无论 Step 1 选择的是本地模式还是 API 模式,产品详情与表现数据始终来源于本地 `data/` 目录下的 Excel 文件。

根据 Step 1 获取到的已触发产品的 `{产品代码}`,前往 `data/product_data.xlsx` 采集该产品离当前日期最近一天的最大回撤数据。
* **数据来源**:`data/product_data.xlsx`
* **逻辑字段属性要求**:
  - 每日最大回撤
  - 单周最大回撤
  - 单月最大回撤
  - 回撤开始日期
  - 回撤持续时间(天数)
  - 数据采集时间

---

### Step 3: 客观输出结论
结合 Step 1 和 Step 2 获取的数据,判定并输出异常结论。
* **分析逻辑**:
  1. 描述异常回撤的具体数据情况。
  2. 根据获取的 `{产品代码}` 与 `{策略分类}`,检索 `data/trigger_rule.xlsx` 中对应的"触发规则",描述产品数据触发的具体规则条件。
  3. **注意**:触发规则可能包含多个条件,需甄别出当前具体命中并超出了哪一个条件(如单日回撤超限或单周回撤超限),并在结论中仅描述异常的情况。

* **输出格式模板**:
  ```markdown
  【{数据日期} 售后提醒】

  - {监测事件名称}

  - 结论:
    {产品名称A}单日最大回撤为 6.2%,超过 5%。
    {产品名称B}单周最大回撤为 5.5%,超过 5%。

  - 明细:
    产品名称 | 单日最大回撤 | 单周最大回撤 | 回撤开始日期 | 回撤持续时间 | 数据采集时间 | 触发状态
    ---|---|---|---|---|---|---
    XXX产品A | 6.2% | - | YYYY-MM-DD | 5天 | YYYY-MM-DD | 已触发
    XXX产品B | - | 5.5% | YYYY-MM-DD | 7天 | YYYY-MM-DD | 已触发
  ```

---

### Step 4: 产品信息解读
根据 Step 3 中输出的异常产品 `{产品代码}`,从 `data/product_detail.xlsx` 及 `references/investment_research/product_reports/` 文件夹获取产品的基本面与持仓信息。

* **⚠️ 降级策略**:如 `product_reports/` 下不存在以 `{产品代码}` 命名的报告文件,自动降级为仅依赖 `product_detail.xlsx` 的基础画像进行分析,并在输出中注明"未检索到该产品专项研报"。

* **数据来源一:`data/product_detail.xlsx` 基础画像**
  - 四大分类归属
  - 策略类型
  - 风险等级
  - 管理人
  - 所属机构
  - 关联大类资产
  - 策略标签

* **数据来源二:`references/investment_research/product_reports/` 目录下以 `{产品代码}` 命名的投资报告**
  - 从报告文本中提取:业绩表现、持仓风格/主题/行业、投资策略、产品相关的市场展望和行业展望,以及类似场景的历史案例和修复时间参考。

---

### Step 5: 深度投研分析 (产品+市场)
依据产品的不同策略分类,匹配特定的分析框架进行深度归因。

> ⚠️ **数据完整性校验**:Agent 在执行本步骤前应检查 `references/investment_research/` 目录下的可用参考资料数量。
> - 如 `market_research/` 为空,应在输出中明确标注"当前市场研报缺失,以下分析基于模型内在知识,建议后续补充最新研报以提升分析质量"。
> - 如 `product_reports/` 下无匹配产品的投研报告,应遵循 Step 4 的降级策略。

* **核心分析逻辑**:
  1. **匹配分析框架**:依据 Step 4 中识别出的 `{策略分类}`,匹配特定的投研分析路径与归因规则。
  2. **研报深度学习**:优先学习 `references/investment_research/market_research/` 文件夹中最新的市场与产品分析内容;其次可去 Wind 数据库中检索相关研报,进行深度归纳和总结。
  3. **调用分析协议**:**必须读取并严格遵循** 本技能包下的静态投研知识库:[evaluate_protocol.md](references/evaluate_protocol.md),按其中定义的 **11 种策略分类归因框架及核对清单** 对产品和市场进行独立分析。
  4. **生成动作建议**:参照 [evaluate_protocol.md](references/evaluate_protocol.md) 的"投资建议决策矩阵"给出对应的加减仓或定投建议。

---

### Step 6: 输出研判内容
经过 Step 5 的多维归因分析,向系统输出投研结论。
* **输出规范**:每个产品独立生成,严禁合并;字数严格控制在 **300 至 350 字** 之间;必须在正文中**明示信息来源与参考研报**。
* **结构化要求**:
  ```text
  【***产品投研分析 (YYYY-MM-DD)】

  1. 现象描述 : [简述市场和产品回撤的数据事实,为其提供量化支持]
  2. 原因分析 : [深入剖析本次触发回撤的核心驱动因子、超额变化及市场成因]
  3. 前瞻判断及建议 : [预测后市表现、产品修复周期,并指明具体的持有/操作建议与关键监测指标]
  ```

---

### Step 7: 输出对客沟通的参考话术
生成可供客群经理直接发送给客户的话术内容,旨在实现售后陪伴、安抚情绪或增强信心。
* **字数限制**:单产品控制在 **200 至 250 字**。若同一客户涉及多只产品,需由 Agent 进行融合拼接,字数可按比例增加。
* **结构规范**:
  - 包含问候语、客观描述对比、原因分析、历史修复周期参照、研究员观点及明确持仓建议。
* **🚨 话术禁用词标准**:生成话术前,**必须读取并遵循** `references/banned_words.yml` 中定义的完整禁用词与专业替代词对照表。严禁在对客话术中使用任何禁用词。

---

### Step 8: 内容推送与工作流状态推进 (整合 watchlist_update)
本步骤实现最终的内容分发与流程状态更新,分为两部分:

#### 1. 飞书卡片推送 (全模式通用)
调用推送脚本安全发送飞书卡片。
* **依赖脚本**:`scripts/feishu_push.py`
* **接收通道**:飞书"产品经理"群
* **执行条件**:仅在 Step 1 存在已触发的事件时执行;若无触发数据,则静默跳过。
* **⚠️ 降级策略**:推送失败自动重试 2 次,仍失败则将结果序列化写入 `reports/push_failed_YYYY-MM-DD.json` 待人工重发。

#### 2. 工作流状态回传 (API 模式专属)
如果是通过 API 查询模式触发且存在有效的事件 ID (`watch_id`),在飞书推送成功后,需运行状态更新脚本以推进流程状态:
* **运行指令**:
  ```bash
  python3 scripts/watchlist_update.py --watch-id {watch_id} --stage 3 --reason "回撤异常监测日报、投研分析及客户陪伴话术已自动生成并发送至飞书群,标记为一线触达完成。"
  ```
* **执行条件**:仅当在 Step 1 中获取到 API 事件 ID (`watch_id`),且非静默/测试模式下运行时执行。
* **⚠️ 降级策略**:若回传失败,程序需在日报中标记 `workflow_status: "pending_manual_retry"` 以备后续人工重试。

## 📁 日报归档与保留规范
* **命名规则**:`reports/YYYY-MM-DD.json`。同一日多次运行时采用覆盖写入策略(以最后一次运行结果为准)。
* **保留周期**:`reports/` 下的日报文件默认保留 90 天。超过保留期的历史日报由运维工具或 Cron 清理任务定期归档。
